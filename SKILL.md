# Teamflow

## 核心设计原则

Teamflow 不是"强制并行框架"，而是"思考策略路由器"。其核心判断：**不是所有任务都适合拆分并行。**

---

## 第一步：任务类型分类（自动判断）

收到 teamflow 请求时，主 agent 必须先判断任务属于哪一类：

### 任务类型矩阵

| 类型 | 特征关键词 | 适合模式 |
|------|-----------|---------|
| **A. 开放性思辨** | 意义、价值、应该、真相、美、灵魂、伦理、哲学、观点、感受 | **Solo** |
| **B. 信息整合** | 分析、对比、综述、报告、收集、整理、多方汇总 | **渐进式并行** |
| **C. 可执行规划** | 开发、设计、实现、构建、修复、创建功能 | **标准teamflow** |
| **D. 简单执行** | 单文件编辑、<50行修改、直接答案、小工具 | **Solo** |

### 分类决策树

```
用户: "使用 teamflow [任务]"
    │
    ▼
检测任务类型
    │
    ├── 含 A 类关键词？（意义/价值/应该/真相/美/灵魂/伦理/哲学/观点）
    │       │
    │       └── 是 → 触发 Solo-Deep 模式（禁用 teamflow 并行）
    │
    ├── 含 B 类关键词？（分析/对比/综述/报告/收集）
    │       │
    │       └── 是 → 触发 渐进式并行
    │
    ├── 含 C 类关键词？（开发/设计/实现/构建/功能）
    │       │
    │       └── 是 → 触发 标准teamflow
    │
    └── 其他 → 评估复杂度指标（文件数/代码量/耦合度）
                    │
                    ├── 简单 → Solo
                    └── 复杂 → Team
```

### A 类任务的特殊处理

对于**开放性思辨任务**（如"思考人生意义"），teamflow 必须：

1. **禁用 prompt-engineer 预框架**——不预设"有哪些视角"
2. **禁用平面并行**——不把思考切分成独立片段
3. **启用 Solo-Deep 模式**——主 agent 自己深度推理
4. **保留完整推理链**——输出中必须包含"为什么这样想"

```
用户: "使用 teamflow 思考人生意义"
    │
    ▼
检测到 A 类关键词（意义）
    │
    ▼
┌─────────────────────────────────────┐
│  Solo-Deep 模式                      │
│  • 不拆分任务                        │
│  • 不预设视角框架                      │
│  • 主 agent 深度推理                  │
│  • 输出推理链条而非结论列表             │
└─────────────────────────────────────┘
```

## 第二步：渐进式并行（适用于 B/C 类任务）

当任务适合 team 模式时，采用**三阶段递进结构**，而非平面并行：

```
Phase 1（并行）：多视角独立探索
    │  Agent-A → 初步结论 A
    │  Agent-B → 初步结论 B
    │  Agent-C → 初步结论 C
    ▼
Phase 2（对抗）：交叉质疑
    │  Agent-D（挑战者）→ 质疑 A/B/C 的漏洞
    │  Agent-A/B/C 回应
    ▼
Phase 3（收敛）：综合洞见
    │  主 agent 整合所有交锋
    │  输出超越原始视角的洞见
    ▼
Final Output
```

### 为什么需要 Phase 2

平面并行的致命缺陷：三个 agent 各自得出结论，然后"礼貌性整合"。没有真正的交锋，就没有超越各自视角的洞见。

Phase 2 的价值：**让结论互相碰撞，在裂缝中诞生真正的洞见。**

## 标准 Teamflow 流程（B/C 类任务）

```
用户: "使用 teamflow [任务]"
    │
    ▼
┌─────────────────────────────────────┐
│ 1. 任务分类（自动判断）              │
│    → Solo-Deep / 渐进式并行 / 标准   │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 2. Prompt-Engineer（仅 B/C 类）     │
│    生成专业提示词，但不预设视角       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 3. 任务拆分 + 批次规划               │
│    → Plan agent                      │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 4. Phase 1 并行探索                  │
│    → Worker-A / B / C（≤3 并发）     │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 5. Phase 2 对抗质疑（可选）          │
│    → 挑战者 agent 驳论               │
│    → 原 agent 回应                   │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 6. Phase 3 收敛整合                  │
│    → Reviewer 最终审查               │
│    → 主 agent 综合洞见                │
└─────────────────────────────────────┘
```

## Concurrency Limits

- **默认硬性限制**：最多 3 个并发 subagent
- **批次调度**：超过 3 个的并行任务，按批次调度
- **Phase 2 可选**：只有当需要深度交锋时启用

---

## Failure Recovery

1. Subagent 失败 → 重试一次
2. 重试失败 → fallback solo（单 agent 接手）
3. Plan 失败 → 降级为直接执行
4. **Solo-Deep 模式中**：若推理卡住 → 可派生单个探索 agent 辅助

---

## Output Contract

### Solo-Deep 模式输出

```markdown
## 深度思考

### 推理链条
[从问题出发，一步步推演的过程，
 不是结论列表，而是思维轨迹]

### 核心洞见
[最终得出的洞察，附带它的来由]

### 可能的争议点
[自己认为最脆弱的地方，供后续讨论]
```

### Team 模式输出

```markdown
## Team Execution Summary

### 任务类型
[A 类：开放性思辨 / B 类：信息整合 / C 类：可执行规划]

### 分工
- [x] Phase 1: @worker-a / @worker-b / @worker-c
- [x] Phase 2: @challenger（对抗质疑）
- [x] Phase 3: 收敛整合

### 推理链条保留
[各 agent 的核心推理路径，而非仅结论]

### Review
[@reviewer 整合的最终洞见]
```

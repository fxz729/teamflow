# Agent Prompts

## Prompt-Engineer（改进版）

**触发**：B/C 类任务，且任务**不是**开放性思辨

**职责**：生成专业提示词，并用生成的提示词**完全替换原始输入**传递给后续步骤

**关键改进**：
- 对于 A 类任务，不生成提示词，直接建议使用 Solo-Deep 模式
- 生成的专业提示词将**完全替换原始用户输入**传递给后续步骤（任务拆分、Worker 执行）

**Prompt 模板**：
```
你是一个提示词工程专家。为以下任务生成专业级提示词。

任务：{user_task}
上下文：{current_context}

**重要**：你的输出将用于**完全替换原始输入**传递给后续执行步骤。
生成的提示词应该是完整的、专业的任务描述，让执行者直接按照生成的提示词执行，无需回看原始输入。

判断：
- 如果这是开放性思辨任务（含"意义/价值/应该/真相/美/灵魂/伦理/哲学/观点"等词）
  → 输出 "SOLO-DEEP" 并说明原因
- 如果这是信息整合/可执行规划任务
  → 继续生成提示词

---

如果是 B/C 类任务，继续：

要求：
1. 生成完整专业任务描述（可直接执行的程度）
2. 指定可验证的验收标准
3. 明确输出格式要求
4. 说明执行策略（但不预设具体视角）

输出格式：

## 任务类型
[B 类: 信息整合 / C 类: 可执行规划]

## 任务理解
[对任务的深度理解]

## 生成的专业提示词
【这部分将完全替换原始输入，传递给后续执行步骤】
[完整的、专业的任务描述，包含：
- 清晰的任务目标
- 可验证的验收标准
- 期望的输出格式
- 执行策略说明
]

## 验收标准
[可量化验证的标准]

## 执行备注
[执行时需要注意的关键点，如有]
```

**执行替换协议**：
```
Prompt-Engineer 输出
    ↓
    ↓ 【提取"## 生成的专业提示词"部分】
    ↓
    ↓ 【完全替换原始 user_task】
    ↓
    → 传递给 Planner（任务拆分）
    → 传递给 Worker（执行）
```

## Solo-Deep 模式（新增）

**触发**：A 类开放性思辨任务

**职责**：主 agent 直接进行深度推理，不拆分，不并行

**Prompt 模板**：
```
你正在进行一次深度思考。

任务：{user_task}

要求：
1. 不拆分任务——这是不可分割的整体思考
2. 不预设框架——不要告诉自己"应该从哪几个角度思考"
3. 保留推理链——输出你的思维轨迹，不只是结论
4. 接受开放结局——不需要强制给出"标准答案"

格式：

## 推理轨迹
[从这个问题出发，你的思维是如何一步步推进的。
 这应该是一段连贯的思维声音，反映你在思考中遇到的每一个转折点。]

## 洞察
[在这个推理过程中，自然涌现的核心洞察]

## 未解之处
[你认为还需要继续思考的地方]

## 可选的外部视角
[如果有某种视角你认为值得探索但你没有深入，可以提及]
```

## Planner

**触发**：B/C 类任务，需要多步骤计划

**职责**：任务拆分和 worker 分派

### Worker 分配规则

#### 分配原则
1. **文件级锁定**：同一文件最多分配给 1 个 worker
2. **依赖优先**：有依赖关系的子任务，分配给同一 worker 或串行执行
3. **专业匹配**：根据子任务的专业领域匹配 worker 角色
4. **负载均衡**：尽量让各 worker 的工作量均衡

#### 分配算法

```
输入：子任务列表 T = [t1, t2, ..., tn]
输入：worker 列表 W = [worker-a, worker-b, worker-c]

1. 分析每个子任务的：
   - 涉及文件（用于检测冲突）
   - 专业领域（用于匹配）
   - 依赖关系（用于排序）

2. 冲突检测：
   for each file in all_task_files:
       if file appears in multiple tasks:
           merge these tasks into single worker

3. 分配执行：
   available_workers = W
   for task in sorted_by_dependencies(T):
       if task.requires_specific_role:
           worker = select_worker_by_role(available_workers, task.role)
       else:
           worker = select_least_loaded_worker(available_workers)
       assign task to worker
       mark worker as busy if at_capacity

4. 验证约束：
   - 每个 worker 最多处理 2 个 write-heavy 任务
   - 同一文件不拆分到不同 worker
```

### Prompt 模板

```
你是一个任务规划专家。基于任务类型，将任务拆分为独立可验证的子任务：

任务：{user_task}
任务类型：{A/B/C}

分析步骤：
1. 识别所有涉及的源文件
2. 识别每个子任务的专业领域
3. 检测文件级冲突（同文件多任务需合并）
4. 识别任务间依赖关系
5. 应用分配算法

要求：
1. 每个子任务独立可执行
2. 最小化共享写入冲突
3. 最多分派 3 个 worker（worker-a, worker-b, worker-c）
4. 标注每个子任务的验收标准
5. 识别任务间的依赖关系
6. 判断是否需要 Phase 2（对抗质疑）

输出格式：

## 任务类型
[B 类: 信息整合 / C 类: 可执行规划]

## 任务拆分
| # | 子任务 | 涉及文件 | 执行者 | 验收标准 | 依赖 |
|---|--------|----------|--------|----------|------|
| 1 | ...    | [文件列表] | worker-a | ... | - |
| 2 | ...    | [文件列表] | worker-b | ... | 依赖#1 |
| 3 | ...    | [文件列表] | worker-c | ... | - |

## 冲突检测结果
[列出检测到的文件冲突及处理方式]

## 是否需要 Phase 2
[是/否。如果是"是"，说明需要挑战什么]

## 执行顺序
[说明哪些可以并行，哪些必须串行]

## 批次规划（如果任务>3）
| 批次 | 包含的子任务 | 并发数 | Barrier 条件 |
|------|-------------|--------|--------------|
| Batch 1 | #1, #2 | 2 | 全部完成 |
| Batch 2 | #3, #4 | 2 | Batch 1 完成 |
```

## 批次调度机制

### Barrier 规则（增强版 2026-04-11）

批次间的 barrier 同步条件：

| Barrier 类型 | 触发条件 | 超时动作 | 超时时间 |
|-------------|---------|----------|----------|
| All-complete | 批次内所有 worker 完成 | 超时则**强制进入下一批次**，未完成 worker 标记为失败 | 5 分钟 |
| 文件锁释放 | 所有文件锁释放 | 超时则**警告并继续**，记录未释放锁 | 2 分钟 |
| Review 完成 | Reviewer 输出完成 | 超时则**跳过 Review**，使用已有结果 | 3 分钟 |

### Barrier 超时处理流程（2026-04-11 新增）

```
当 Barrier 超时时：
1. 记录超时日志
   - barrier_type: [All-complete/FileLock/Review]
   - timeout_at: [时间戳]
   - pending_agents: [未完成的 agent 列表]

2. 执行超时动作
   - All-complete: 强制进入下一批次，未完成 agent 标记失败
   - FileLock: 警告并继续，记录锁状态
   - Review: 跳过 Review，继续收敛

3. 通知主 agent
   - 发送超时事件
   - 说明已采取的动作
   - 传递当前状态
```

### 批次超时

```
批次执行超时 = max(各子任务超时) + 2分钟缓冲
默认批次超时 = 5 分钟
批次总数超时 = 15 分钟
```

### 批次执行伪代码

```
batches = partition_tasks(tasks, max_size=3)
for batch in batches:
    start_time = now()

    # 并行执行批次内任务
    parallel_execute(batch.workers)

    # 等待 barrier 条件
    wait_for_barrier(batch, timeout=batch_timeout)

    if barrier_timeout:
        log_warning(f"Batch {batch.id} barrier timeout")
        record_timeout_event(batch)  # 2026-04-11 新增
        force_proceed_to_next_batch()  # 强制进入下一批次

# 如果批次总数超时
if total_timeout:
    force_convergence()  # 强制收敛
```

## Phase 2: 挑战者（增强版 2026-04-11）

**触发**：需要深度交锋的 B/C 类任务

**职责**：质疑其他 agent 的结论，迫使他们回应

**Prompt 模板**：
```
你是对抗性思考专家。你的任务是质疑其他 agent 的结论。

原始任务：{user_task}

Phase 1 结论：
{agent_a_conclusion}
{agent_b_conclusion}
{agent_c_conclusion}

你的任务：
1. 找出每个结论中最脆弱的假设
2. 提出尖锐的质疑，不是为了否定，而是为了让思考更深入
3. 识别结论之间的矛盾或未覆盖的区域

### 质疑要求（2026-04-11 增强）
1. **必须引用 Phase 1 输出的具体内容**
   - 不能泛泛而谈，必须引用原话
   - 例如："你得出'系统性能优秀'的结论，但报告中说平均响应时间是 800ms，请解释这个矛盾"

2. **指出该内容的前提假设**
   - 明确说出假设是什么
   - 例如："你假设所有用户都有高速网络，但未考虑移动用户的弱网环境"

3. **说明该假设可能失败的条件**
   - 具体描述边界情况
   - 例如："当并发用户超过 1000 时，这个假设会失效"

4. **提出替代方案或遗漏的维度**
   - 给出具体建议
   - 例如："建议补充弱网环境下的性能测试"

### 禁止的行为
- ❌ 泛泛而谈的"假设前提"
- ❌ 没有具体针对性的"潜在风险"
- ❌ 只说"有问题"而不说具体是什么问题

质疑格式：
## 对 @worker-a 的质疑
[必须引用 worker-a 的具体输出内容]

## 对 @worker-b 的质疑
[必须引用 worker-b 的具体输出内容]

## 对 @worker-c 的质疑
[必须引用 worker-c 的具体输出内容]

## 结论间的矛盾
[如果有的话]
```

## Reviewer

**触发**：所有子任务完成后，需要最终审查

**Prompt 模板**：
```
你是审查专家。审查团队执行结果：

任务：{original_task}
任务类型：{A/B/C}

Phase 1 输出：
{worker_outputs}

Phase 2 质疑（如果有）：
{challenger_output}

审查要点：
1. 是否满足原始需求
2. 子任务间是否有冲突
3. 推理链条是否完整
4. Phase 2 的质疑是否得到了回应

输出格式：
## 审查结果
✅ 通过 / ⚠️ 需要补充 / ❌ 需要修改

## 整合洞察
[将所有输出综合成超越原始视角的核心洞见]

## 推理链条摘要
[从原始问题到最终洞见的完整推理路径]
```

## Role Mapping to Claude CLI Agents

| Role | Claude CLI Agent | 职责 |
|------|-----------------|------|
| `solo-deep` | 主 agent | 开放性思辨任务，深度推理 |
| `prompt-engineer` | subagent（自动派生） | B/C 类任务生成提示词，A 类建议用 solo |
| `planner` | `Plan` agent | 任务拆分 + 批次规划 |
| `worker` | 直接执行 | 执行分配的子任务，输出推理链 |
| `challenger` | subagent（自动派生） | Phase 2 对抗质疑 |
| `reviewer` | `code-reviewer` agent | 最终审查整合 |

# Routing Rules

> **实现说明**：本文件的分类逻辑已在 `../lib/routing_rules.py` 中实现为可执行代码。
> 文档用于理解规则，代码用于实际执行和测试。



## 任务类型分类体系

### 四类任务定义

#### A 类：开放性思辨（Open-Ended Reasoning）
**定义**：没有标准答案，需要深度推理和价值判断的任务。

**触发关键词（显式）**：
- 意义、价值、应该、真相、美、灵魂
- 伦理、道德、哲学、观点、感受
- 判断、评价、好不好、如何看待
- 人生、存在、本质、超越

**触发关键词（隐式 - 需要结合语境判断）**：
- 思考、反思、探讨、探索（单独使用且无具体技术上下文时）
- "如何看待..."、"如何评价..."、"如果是你..."
- "最好的方式"、"最优解"（在价值观上下文中）
- 涉及情感、审美、个人偏好的问题

**不触发 A 类的场景**：
- "思考怎么实现这个功能" → C 类（具体技术问题）
- "反思项目结构" → C 类（有明确技术上下文）
- "探索多种方案" → B 类（有技术评估上下文）

**"思考/反思"上下文检测规则（2026-04-11 修复）**：
当"思考/反思"出现在句中时，需结合后续词语判断：
- "思考/反思 + 具体技术名词（架构、代码、系统、项目、模块、接口）" → C 类
- "思考/反思 + 抽象概念（意义、价值、人生、存在、本质、灵魂）" → A 类

**修复验证**：
| 输入 | 原判定 | 修复后 | 说明 |
|------|--------|--------|------|
| "思考架构" | C 类 | C 类 | "架构"是技术上下文 |
| "反思项目结构" | C 类 | C 类 | "项目结构"是技术上下文 |
| "思考人生意义" | A 类 | A 类 | "人生意义"是抽象概念 |
| "反思存在本质" | A 类 | A 类 | "存在本质"是抽象概念 |

**路由决策**：**Solo-Deep 强制模式**
- 禁用 prompt-engineer（不预设视角）
- 禁用平面并行（不拆分思考）
- 主 agent 深度推理，保留推理链

---

#### B 类：信息整合（Information Integration）
**定义**：从多个来源/角度收集信息并整合的任务。

**触发关键词**：
- 分析、对比、综述、报告
- 收集、整理、汇总、多方
- 研究、调查、评估

**路由决策**：**渐进式并行**
- Phase 1：多源并行探索
- Phase 2：对抗质疑（可选）
- Phase 3：收敛整合

---

#### C 类：可执行规划（Executable Planning）
**定义**：有明确可交付物的工程/开发任务。

**触发关键词**：
- 开发、设计、实现、构建
- 创建功能、修复 bug、重构
- 架构、部署、测试

**路由决策**：**标准 teamflow**
- Plan agent 任务拆分
- Worker 并行执行
- Reviewer 最终审查

---

#### D 类：简单执行（Simple Execution）
**定义**：可一次性完成的小型任务。

**触发条件**（满足任一）：
- 单文件编辑
- <50 行代码修改
- 直接答案查询
- 小型工具创建

**路由决策**：**Solo 快速模式**

## Solo vs Team Decision Matrix

### 量化复杂度阈值

| 指标 | Solo 阈值 | Team 阈值 | 测量方法 |
|------|-----------|-----------|----------|
| 涉及文件数 | ≤ 1 | ≥ 2 | `Glob` 扫描或人工估算 |
| 预估代码行数 | < 50 | ≥ 50 | 静态分析或人工估算 |
| 子任务数 | 1 | ≥ 2 | Plan agent 拆分结果 |
| 专业领域数 | 1 | ≥ 2 | 任务描述分析 |
| 依赖深度 | 0 | ≥ 1 | 文件 import 分析 |

### 复杂度评分公式

```
复杂度得分 =
  (文件数得分 × 0.3) +
  (代码行数得分 × 0.3) +
  (任务数得分 × 0.2) +
  (领域数得分 × 0.2)

得分解释：
- 0.0 - 0.3 → Solo
- 0.4 - 0.6 → 渐进式并行（B 类）
- 0.7 - 1.0 → 标准 teamflow（C 类）
```

**维度得分规则**：
- 文件数：≤1 → 0, 2-3 → 1, >3 → 2
- 代码行数：<50 → 0, 50-200 → 1, >200 → 2
- 子任务数：1 → 0, 2-4 → 1, >4 → 2
- 领域数：1 → 0, 2 → 1, >2 → 2

### 任务规模上限

| 规模 | 子任务数 | 处理方式 |
|------|----------|----------|
| 小型 | < 5 | 标准 teamflow |
| 中型 | 5 - 15 | 精简 teamflow（跳过可选阶段） |
| 大型 | 15 - 30 | writing-plans + 分阶段执行 |
| 超大型 | > 30 | **拒绝执行**，提示用户拆分 |

### Solo vs Team 最终决策

| 条件 | 路由决策 |
|------|----------|
| A 类关键词命中 | Solo-Deep |
| D 类条件全部满足（文件≤1，代码<50行） | Solo |
| 复杂度得分 ≤ 0.3 | Solo |
| B 类关键词命中，且复杂度得分 0.4 - 0.6 | 渐进式并行 |
| C 类关键词命中，或复杂度得分 ≥ 0.7 | 标准 teamflow |
| 多个类型关键词共存 | 按 SKILL.md 中的多标签评分机制 |

## Decision Flow（完整版）

```
用户输入: "使用 teamflow [任务]"
    │
    ▼
┌─────────────────────────────────────┐
│ Step 1: 任务类型分类                │
│ 检测关键词或评估复杂度               │
└─────────────────────────────────────┘
    │
    ├── A 类关键词命中
    │       │
    │       ▼
    │   ┌─────────────────────────────────────┐
    │   │ Solo-Deep 模式                      │
    │   │ • 不调用 prompt-engineer             │
    │   │ • 不拆分任务                         │
    │   │ • 主 agent 深度推理                  │
    │   │ • 输出完整推理链                     │
    │   └─────────────────────────────────────┘
    │
    ├── B 类关键词命中
    │       │
    │       ▼
    │   渐进式并行（Phase 1 → 2 → 3）
    │
    ├── C 类关键词命中
    │       │
    │       ▼
    │   标准 teamflow（Plan → Worker → Review）
    │
    └── 其他
            │
            ▼
        评估复杂度指标
            │
            ├── 简单（单文件/<50行）→ Solo
            └── 复杂 → 标准 teamflow
```

## Concurrency Guidelines

### 默认：3 并发

```markdown
任务数 ≤ 3 → 全并行
任务数 > 3 → 分批调度（每批 ≤ 3）
```

### 渐进式并行特殊规则

```
Phase 1: 最多 3 个探索 agent 并行
Phase 2: 1 个挑战者 agent + 被挑战 agent 回应（串行为主）
Phase 3: 收敛阶段（主 agent 执行，非 subagent）
```

## Solo-Deep 模式规则

### 何时触发

- A 类任务（开放性思辨）
- 用户请求"思考..."、"反思..."、"探讨..."
- 任务描述含"意义"、"价值"、"应该"等关键词

### 核心约束

1. **不预设框架**：不调用 prompt-engineer 生成视角列表
2. **不拆分任务**：不把"人生意义"拆成"存在主义视角"+"东方哲学视角"
3. **保留推理链**：输出包含"为什么这样想"，而非仅"想了什么"
4. **接受开放结局**：不需要强制收敛到"标准答案"

### 输出要求

```
## 深度思考

### 推理轨迹
[从问题出发，一步步推演的过程。
 这是思维的声音，不是结论的罗列。]

### 核心洞察
[在这个推理过程中诞生的洞见]

### 未解之处
[自己认为还需要继续思考的地方]
```

## Fallback Rules

| 失败场景 | 应对策略 |
|---------|---------|
| subagent 失败 | 重试一次，失败则 solo 接手该任务 |
| Plan agent 失败 | 降级为直接执行（主 agent自己做） |
| Phase 2 挑战者无效 | 跳过 Phase 2，直接进入收敛 |
| 团队模式整体失败 | 回退到 Solo-Deep 模式 |

## 场景示例

### B 类场景示例（信息整合）

**示例 1：竞品分析**
```
输入："使用 teamflow 分析竞品 A 和竞品 B 的差异化策略"
关键词：分析、竞品、策略
分类：B 类（信息整合）
路由：渐进式并行
```

**示例 2：多方资料收集**
```
输入："使用 teamflow 收集并整理关于 AI 安全的最新研究"
关键词：收集、整理、研究
分类：B 类（信息整合）
路由：渐进式并行
```

**示例 3：多策略对比**
```
输入："使用 teamflow 对比分析四种心理创伤疗愈方法的优劣"
关键词：对比、分析、疗愈方法
分类：B 类（信息整合）
路由：渐进式并行
```

**示例 1：功能开发**
```
输入："使用 teamflow 为电商系统添加用户评论功能"
关键词：添加、开发、功能
分类：C 类（可执行规划）
路由：标准 teamflow
```

**示例 2：系统设计**
```
输入："使用 teamflow 设计一个任务管理系统的数据模型"
关键词：设计、任务管理系统
分类：C 类（可执行规划）
路由：标准 teamflow

**示例 3：架构构建**
```
输入："使用 teamflow 构建一个微服务架构，包含用户、订单、支付服务"

分类：C 类（可执行规划）

### 多标签共存示例

**示例：B+C 共存**
```
输入："使用 teamflow 分析现有代码库并设计重构方案"
关键词：分析（B类）、设计/重构（C类）
分类：B 类关键词 1 个，C 类关键词 1 个，数量相等
路由：标准 teamflow（相等时默认选择 C 类规则）
```

**示例：A+B+C 全部共存**
```
输入："使用 teamflow 思考这个系统的价值，并分析竞品数据"
关键词：思考（A类）、价值（A类）、分析（B类）
分类：A 类命中 2 个
路由：Solo-Deep（A 类优先级最高）
```

**示例：A+C 共存**
```
输入："使用 teamflow 思考如何实现一个更好的认证系统"
关键词：思考（A类）、实现/系统（C类）
分类：A 类命中 1 个
路由：Solo-Deep（A 类优先级最高）
```

### C 类否定示例（避免过度分类）

| 输入 | 实际分类 | 原因 |
|------|---------|------|
| `"使用 teamflow 分析代码"` | B 类 | "分析"是 B 类关键词 |
| `"使用 teamflow 思考架构"` | C 类 | "架构"是技术上下文，触发 C 类 |
| `"使用 teamflow 修复 bug"` | D 类 | 单文件修复属于 D 类 |
| `"使用 teamflow 重构这个模块"` | C 类 | 重构属于 C 类（开发/构建/功能）|

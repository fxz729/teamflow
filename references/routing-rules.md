# Routing Rules

## 任务类型分类体系

### 四类任务定义

#### A 类：开放性思辨（Open-Ended Reasoning）
**定义**：没有标准答案，需要深度推理和价值判断的任务。

**触发关键词**：
- 意义、价值、应该、真相、美、灵魂
- 伦理、道德、哲学、观点、感受
- 判断、评价、好不好、如何看待
- 人生、存在、本质、超越

**路由决策**：**Solo-Deep 强制模式**
- 禁用 prompt-engineer（不预设视角）
- 禁用平面并行（不拆分思考）
- 主 agent 深度推理，保留推理链

---

#### B 类：信息整合（Information Integration）
**定义**：从多个来源/角度收集信息并整合的任务。

**触发关键词**：
- 分析、对比、综述、报告
- 收集、整理、汇总、多方
- 研究、调查、评估

**路由决策**：**渐进式并行**
- Phase 1：多源并行探索
- Phase 2：对抗质疑（可选）
- Phase 3：收敛整合

---

#### C 类：可执行规划（Executable Planning）
**定义**：有明确可交付物的工程/开发任务。

**触发关键词**：
- 开发、设计、实现、构建
- 创建功能、修复 bug、重构
- 架构、部署、测试

**路由决策**：**标准 teamflow**
- Plan agent 任务拆分
- Worker 并行执行
- Reviewer 最终审查

---

#### D 类：简单执行（Simple Execution）
**定义**：可一次性完成的小型任务。

**触发条件**（满足任一）：
- 单文件编辑
- <50 行代码修改
- 直接答案查询
- 小型工具创建

**路由决策**：**Solo 快速模式**

---

## Solo vs Team Decision Matrix

### Solo Indicators（A 类自动命中）

| 指标 | 阈值 | 类型 |
|------|------|------|
| 任务类型 | 含 A 类关键词 | A 类 → Solo-Deep |
| 文件数 | 1 | D 类 → Solo |
| 代码行数 | <50 | D 类 → Solo |
| 耦合度 | 高 | D 类 → Solo |
| 输出类型 | 思辨/判断 | A 类 → Solo-Deep |

### Team Indicators（B/C 类）

| 指标 | 阈值 | 说明 |
|------|------|------|
| 文件数 | >3 | 多文件可并行 |
| 任务数 | ≥3 | 明确可拆分 |
| 专业领域 | 多 | 需要多视角 |
| 冲突风险 | 低 | 文件修改不重叠 |
| 验证独立性 | 高 | 子任务可独立测试 |
| 任务类型 | B/C 类 | 信息整合/可执行规划 |

## Decision Flow（完整版）

```
用户输入: "使用 teamflow [任务]"
    │
    ▼
┌─────────────────────────────────────┐
│ Step 1: 任务类型分类                │
│ 检测关键词或评估复杂度               │
└─────────────────────────────────────┘
    │
    ├── A 类关键词命中
    │       │
    │       ▼
    │   ┌─────────────────────────────────────┐
    │   │ Solo-Deep 模式                      │
    │   │ • 不调用 prompt-engineer             │
    │   │ • 不拆分任务                         │
    │   │ • 主 agent 深度推理                  │
    │   │ • 输出完整推理链                     │
    │   └─────────────────────────────────────┘
    │
    ├── B 类关键词命中
    │       │
    │       ▼
    │   渐进式并行（Phase 1 → 2 → 3）
    │
    ├── C 类关键词命中
    │       │
    │       ▼
    │   标准 teamflow（Plan → Worker → Review）
    │
    └── 其他
            │
            ▼
        评估复杂度指标
            │
            ├── 简单（单文件/<50行）→ Solo
            └── 复杂 → 标准 teamflow
```

## Concurrency Guidelines

### 默认：3 并发

```markdown
任务数 ≤ 3 → 全并行
任务数 > 3 → 分批调度（每批 ≤ 3）
```

### 渐进式并行特殊规则

```
Phase 1: 最多 3 个探索 agent 并行
Phase 2: 1 个挑战者 agent + 被挑战 agent 回应（串行为主）
Phase 3: 收敛阶段（主 agent 执行，非 subagent）
```

## Solo-Deep 模式规则

### 何时触发

- A 类任务（开放性思辨）
- 用户请求"思考..."、"反思..."、"探讨..."
- 任务描述含"意义"、"价值"、"应该"等关键词

### 核心约束

1. **不预设框架**：不调用 prompt-engineer 生成视角列表
2. **不拆分任务**：不把"人生意义"拆成"存在主义视角"+"东方哲学视角"
3. **保留推理链**：输出包含"为什么这样想"，而非仅"想了什么"
4. **接受开放结局**：不需要强制收敛到"标准答案"

### 输出要求

```
## 深度思考

### 推理轨迹
[从问题出发，一步步推演的过程。
 这是思维的声音，不是结论的罗列。]

### 核心洞察
[在这个推理过程中诞生的洞见]

### 未解之处
[自己认为还需要继续思考的地方]
```

## Fallback Rules

| 失败场景 | 应对策略 |
|---------|---------|
| subagent 失败 | 重试一次，失败则 solo 接手该任务 |
| Plan agent 失败 | 降级为直接执行（主 agent自己做） |
| Phase 2 挑战者无效 | 跳过 Phase 2，直接进入收敛 |
| 团队模式整体失败 | 回退到 Solo-Deep 模式 |
